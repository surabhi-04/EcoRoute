Overview Section 1 of 4

EcoRoute Enterprise Core Intelligence Layer

As corporate entities navigate aggressive environmental metrics under modern reporting mandates (such as CSRD and ESG initiatives), manual systems break down. EcoRoute addresses this critical demand directly: it bridges operational logistics tracking with strict, enterprise-grade carbon compliance audit trails.

The Spreadsheet & AI Dilemma

Conventional platforms rely on fragile spreadsheet macros and legacy tools prone to leakage, or modern AI agents that prone to hallucinations. Legally, neither can provide the guarantees corporate lawyers and government inspectors require.

The EcoRoute Differentiator

EcoRoute implements a **100% deterministic, mathematically auditable pipeline**. By pairing rigid graph-based distance lookups (Dijkstra) with transactional ring-fencing, audit logs are fully immutable and compliant with international standards.

Architecture Section 2 of 4

Multi-Tenant Schema Separation & Dual-Persistence Architecture

To scale gracefully while supporting competitors securely, EcoRoute operates a dual-layer persistence system built on Java Spring Boot and MySQL:

A

Multi-Tenant Ring-Fencing

Competing networks (e.g. DHL and FedEx Express) run on shared server instances but are completely isolated at the query tier. Spring Security intercepts the principal context on each request and attaches a strict, non-bypassable `company_id` filter to all CRUD statements.

B

Split-Persistence Execution

The persistence tier splits workloads dynamically: **Hibernate JPA** manages complex, transactional domain graphs (Users, Companies, Audit Logs, Shipments) with reliable transaction boundaries, while **Spring JdbcTemplate** bypasses ORM overhead to process raw database lookups for coordinate pairs instantly.

Security Roles Section 3 of 4

Operational Account Lifecycles & Security Constraints

User onboarding and data visibility rules are tightly governed by a native role hierarchy to guarantee operational security:

System Administrator

Manages the tenant onboarding queue. Reviews self-registered corporate applications in `PENDING_APPROVAL` status and activates them. No access to logistical shipments.

Logistics Manager

Acts as tenant account owner. Operates shipment entry gates, maps cargo weights, processes route coordinates, and provisions/manages local team members.

Environmental Auditor

Possesses full read-only visibility over their tenant's logistics logs. Performs audits and generates legally certifiable compliance papers via print engine layouts.

MySQL Initialization (schema.sql) 
-- =========================================================================
-- EcoRoute Enterprise Multi-Tenant DDL Schema Initialization Script
-- Database Target: MySQL 8.0+
-- =========================================================================

CREATE DATABASE IF NOT EXISTS ecoroute DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
USE ecoroute;

-- 1. Companies Table (Tenant Entities)
CREATE TABLE IF NOT EXISTS companies (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(150) NOT NULL UNIQUE,
    industry_sector VARCHAR(100) NOT NULL,
    onboarding_status VARCHAR(50) NOT NULL DEFAULT 'PENDING_APPROVAL',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_company_status (onboarding_status)
) ENGINE=InnoDB;

-- 2. Users Table (Sub-Users isolated by Company Relationship)
CREATE TABLE IF NOT EXISTS users (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50) NOT NULL UNIQUE,
    password VARCHAR(100) NOT NULL,
    email VARCHAR(100) NOT NULL,
    role VARCHAR(50) NOT NULL,
    company_id BIGINT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (company_id) REFERENCES companies(id) ON DELETE CASCADE,
    INDEX idx_user_company (company_id)
) ENGINE=InnoDB;

-- 3. Distance Lookup Matrix Table (Spatial Coordinate Routing cache)
CREATE TABLE IF NOT EXISTS distance_lookups (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    city_a VARCHAR(100) NOT NULL,
    city_b VARCHAR(100) NOT NULL,
    distance_km DOUBLE NOT NULL,
    UNIQUE KEY uq_city_pair (city_a, city_b),
    INDEX idx_cities (city_a, city_b)
) ENGINE=InnoDB;

-- 4. Shipments Table (Isolated logistics logs)
CREATE TABLE IF NOT EXISTS shipments (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    tracking_number VARCHAR(100) NOT NULL UNIQUE,
    weight_tons DOUBLE NOT NULL,
    transport_mode VARCHAR(50) NOT NULL,
    origin_city VARCHAR(100) NOT NULL,
    destination_city VARCHAR(100) NOT NULL,
    co2_emissions DOUBLE NOT NULL,
    company_id BIGINT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (company_id) REFERENCES companies(id) ON DELETE CASCADE,
    INDEX idx_shipment_company (company_id),
    INDEX idx_shipment_tracking (tracking_number)
) ENGINE=InnoDB;

-- 5. Platform Audit Logs Table (Independent verification trails)
CREATE TABLE IF NOT EXISTS audit_logs (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    action VARCHAR(100) NOT NULL,
    details TEXT NOT NULL,
    username VARCHAR(100) NOT NULL,
    company_name VARCHAR(150) NOT NULL,
    INDEX idx_audit_timestamp (timestamp),
    INDEX idx_audit_company (company_name)
) ENGINE=InnoDB;
